# 大模型驱动的机械臂智能执行平台 — 技术方案概览

> 版本：v1.1 | 2026-03-13

---

## 一、平台定位

本平台是一套 **"感知-决策-执行-学习" 闭环的机械臂智能基础设施**，以 VLA（Vision-Language-Action）、VLM（Vision-Language Model）等大模型为核心引擎，驱动机械臂在不同场景下自主完成操作任务。

核心理念：**不为每个任务写死规则，而是让大模型在场景中自主构建规则、持续优化策略。**

```
┌─────────────────────────────────────────────────────────┐
│                  场景层 (Scenario)                       │
│   物料分拣 │ 精密装配 │ 质检搬运 │ ... 可扩展             │
├─────────────────────────────────────────────────────────┤
│               策略引擎 (Policy Engine)                   │
│   VLM 视觉决策 │ VLA 动作生成 │ LLM 规则推理             │
│   计划缓存 │ 技能库 │ 异常处理                            │
├─────────────────────────────────────────────────────────┤
│              数据与学习层 (Data & Learning)               │
│   Case 自动采集 │ 大模型分析 │ 策略微调 │ 模型微调          │
├─────────────────────────────────────────────────────────┤
│               硬件抽象层 (Hardware Abstraction)           │
│   机械臂控制 │ 视觉传感器 │ 力/力矩传感 │ 边缘计算         │
└─────────────────────────────────────────────────────────┘
```

---

## 二、要解决的问题

传统机械臂方案依赖人工编写固定规则和路径点，面临以下痛点：

| 痛点 | 传统方案 | 本平台方案 |
|------|----------|------------|
| 新场景部署周期长 | 每个场景重新编程，数周调试 | 自然语言描述任务，大模型自动构建逻辑 |
| 环境变化难适应 | 物体位置/姿态变化需重新标定 | VLM 实时理解场景，动态调整决策 |
| 异常处理依赖人工 | 抓取失败需人工干预 | 闭环观察 + 自动修正 |
| 经验难以沉淀 | 调参经验在工程师个人手中 | Case 自动采集 → 大模型分析 → 技能库沉淀 |
| 迁移成本高 | A 场景的方案无法用于 B 场景 | 统一架构，新场景只需配置不需重开发 |
| 力控不灵活 | 固定力阈值，不同物体需反复调参 | 大模型根据物体材质/形状自适应输出力控策略 |

## 三、核心能力

### 3.1 视觉理解（VLM）

大模型直接"看"摄像头画面，理解场景中有什么、在哪里、处于什么状态：

- **场景理解**：识别物体种类、材质、颜色、大小、空间关系
- **目标定位**：Visual Grounding，输出目标 Bounding Box / 6DoF 位姿
- **状态判断**：判断抓取是否成功、物体是否到位、是否存在异常
- **2D→3D 映射**：结合 RGB-D 深度相机将图像坐标转换为机械臂工作空间坐标

### 3.2 智能决策（VLM + LLM）

| 能力 | 说明 |
|------|------|
| **任务规划** | 将自然语言指令（如"把蓝色瓶子放到左边箱子"）拆解为可执行的子任务序列 |
| **逐步决策** | 每一步拍照观察 → VLM 判断当前状态 → 输出下一个动作 |
| **标准动作协议** | 统一 JSON 格式：`move_above` / `lower` / `grasp` / `lift` / `release` / `done` |
| **力控策略推理** | 根据目标物体的材质和形状，自动推理抓取力度、速度曲线和接近角度 |
| **计划模板缓存** | 首次任务由 VLM 逐步推理；成功后缓存为模板，后续同类任务跳过推理直接复用 |
| **异常自修复** | 执行结果不符合预期时自动重新观察、调整策略 |

### 3.3 动作执行（VLA）

端到端的 Vision-Language-Action 模型，直接从图像和语言指令生成关节级动作：

- **SmolVLA**：轻量级 VLA 模型，支持边缘设备部署
- **ACT（Action Chunking with Transformers）**：高精度模仿学习模型，一次预测未来 k 步轨迹
- **训练管线**：支持从示教数据采集到模型微调的完整流程，可对接工业级训练基础设施
- **力矩输出**：VLA 模型不仅输出位置轨迹，还可输出各关节力矩参考值

### 3.4 自学习闭环

```
┌──────────┐     ┌───────────┐     ┌───────────┐     ┌──────────┐
│ 任务执行  │ ──▶ │ Case 采集  │ ──▶ │ 大模型分析 │ ──▶ │ 策略优化  │
└──────────┘     └───────────┘     └───────────┘     └────┬─────┘
      ▲                                                    │
      │              ┌────────────┐                        │
      └────────────  │  技能库更新  │  ◀────────────────────┘
                     └────────────┘
```

| 阶段 | 做什么 | 怎么做 |
|------|--------|--------|
| **采集** | 每次执行自动保存图像序列、动作日志、力传感数据、执行结果 | API 服务的 `/feedback` 接口自动记录 |
| **分析** | 定期将失败/异常 case 批量提交给 LLM 分析 | 调用云端大模型 API 批量处理 |
| **策略调整** | 根据分析结论更新决策规则、力控参数、识别阈值 | Prompt 优化 / 配置热更新 |
| **模型微调** | 累积足够数据后对 VLA 模型做 fine-tuning | 自研训练管线，支持工业级数据规模 |
| **技能沉淀** | 成熟方案封装为 Skill，支持跨场景复用 | 技能库 + 模板缓存系统 |

### 4.2 MCP 服务化架构

平台采用 **MCP（Model Context Protocol）** 作为 AI 大脑与各子系统的标准通信协议。每个能力模块封装为独立的 MCP Server，大模型通过统一的工具调用接口与之交互，实现模型可插拔、模块可替换。

```
                    ┌────────────────────┐
                    │    AI 大脑 (LLM)    │
                    │  Qwen / GPT / Claude│
                    └─────────┬──────────┘
                              │ MCP 协议
            ┌─────────────────┼─────────────────┐
            │                 │                 │
   ┌────────▼──────┐  ┌──────▼───────┐  ┌──────▼───────┐
   │ MCP: 模板缓存  │  │ MCP: 视觉感知 │  │ MCP: 机械臂  │
   │ 精确/向量匹配  │  │ 拍照 + 检测   │  │ 运动 + 力控   │
   └───────────────┘  └──────────────┘  └──────────────┘
            │
   ┌────────▼──────┐
   │ MCP: 数据采集  │
   │ Case 记录+分析 │
   └───────────────┘
```

**MCP Server 列表：**

| MCP Server | 暴露的工具 | 职责 |
|------------|-----------|------|
| **模板缓存** | `query_template`、`save_template` | 根据物料类型/场景匹配已有方案，存储新方案 |
| **视觉感知** | `capture_image`、`detect_objects`、`get_depth` | 图像采集、目标检测与定位、深度信息获取 |
| **机械臂控制** | `move_to`、`grasp`、`release`、`read_force` | 运动控制、力控操作、传感器读取 |
| **数据采集** | `log_case`、`query_failures` | 记录执行过程、查询失败案例 |

> MCP 协议的关键优势：**模型可插拔**。今天用 Qwen，明天换 GPT-4o 或 Claude，MCP Server 不用改动。对客户来说意味着不锁定任何一家 AI 供应商。

### 4.3 三级加速决策机制

为解决云端大模型调用延迟（3–5s）在生产线节拍中不可接受的问题，平台设计了 **三级决策加速架构**，逐级 fallback：

```
请求到达
    │
    ▼
Level 1: 模板精确匹配 ─── 命中 ──→ 直接执行（< 50ms）
    │ 未命中
    ▼
Level 2: 向量相似匹配 ─── 命中 ──→ 微调参数后执行（< 200ms）
    │ 未命中
    ▼
Level 3: 边缘小模型推理 ── 有信心 ─→ 执行（200-500ms）
    │ 信心不足
    ▼
Level 4: 云端大模型推理 ─────────→ 执行（3-5s）+ 结果存入缓存
```

| 级别 | 机制 | 响应时间 | 适用场景 |
|------|------|----------|----------|
| **L1 精确匹配** | 物料类型 + 场景特征 → 直接命中缓存 | **< 50 ms** | 已见过的物料，完全一致 |
| **L2 向量相似** | 场景 Embedding 相似度检索最近模板 | **< 200 ms** | 同类物料有变化（换尺寸、换颜色） |
| **L3 边缘小模型** | Jetson 本地部署轻量 VLM（如 Qwen-VL-2B 量化） | **200–500 ms** | 新物料但场景简单，小模型能判断 |
| **L4 云端大模型** | 调用 Qwen-Max / GPT-4o 等完整推理 | **3–5 s** | 全新场景或复杂异常，需要完整推理能力 |

> 生产线稳态运行时 90%+ 的请求命中 L1/L2，平均决策延迟 **< 100ms**。仅在首次遇到新品类或异常情况时才 fallback 到 L3/L4，且推理结果会自动回写缓存，后续不再触发。

## 五、性能指标

### 5.1 响应时延

得益于三级加速决策机制，系统在不同阶段的时延表现：

| 决策级别 | 端到端时延 | 占比（稳态） | 说明 |
|----------|-----------|-------------|------|
| L1 精确匹配 | **< 50 ms** | ~70% | 已见过的物料直接命中模板 |
| L2 向量相似 | **< 200 ms** | ~20% | 同类物料微调参数后执行 |
| L3 边缘小模型 | **200–500 ms** | ~8% | 本地轻量 VLM 推理 |
| L4 云端大模型 | **3–5 s** | ~2% | 全新场景完整推理，结果回写缓存 |
| **稳态加权平均** | **< 100 ms** | — | 90%+ 请求命中 L1/L2 |

> 产线冷启动时，每种新品类首次走 L4 完整推理（3–5s），之后自动缓存。品类稳定后日常运行几乎全部命中 L1/L2，平均决策延迟 **< 100ms**。

### 5.2 力控能力

| 指标 | 参数 | 说明 |
|------|------|------|
| 力传感精度 | ±0.1 N | 六维力/力矩传感器 |
| 力控频率 | 500–1000 Hz | 闭环力控制周期 |
| 抓取力范围 | 0.5–50 N | 覆盖鸡蛋（~2N）到金属零件（~40N） |
| 力控模式 | 位置/力混合控制 | 支持自由空间位置控制 + 接触面力控制 |
| 碰撞检测 | < 5 ms 响应 | 异常外力触发安全停止 |
| 柔顺控制 | 阻抗/导纳可调 | 适应不同刚度物体的接触操作 |

**大模型自适应力控策略：**

VLM 根据识别到的物体材质自动匹配力控参数：

| 物体类型 | 抓取力 | 接近速度 | 柔顺等级 | 示例 |
|----------|--------|----------|----------|------|
| 脆弱/易碎 | 1–3 N | 慢 (≤30 mm/s) | 高柔顺 | 玻璃瓶、鸡蛋、芯片 |
| 普通/塑料 | 5–15 N | 中 (≤80 mm/s) | 中柔顺 | 塑料件、包装盒 |
| 坚硬/金属 | 15–40 N | 快 (≤150 mm/s) | 低柔顺 | 金属零件、工具 |
| 柔性/可变形 | 2–8 N | 慢 (≤50 mm/s) | 高柔顺 | 织物、线缆、软管 |

> 首次操作新类型物体时由 VLM 推理力控策略，成功后自动沉淀到技能库，后续直接复用。

### 5.3 定位精度

| 指标 | 参数 |
|------|------|
| 视觉定位精度（2D） | ±2 px（VLM Grounding） |
| 空间定位精度（3D） | ±1–3 mm（RGB-D 反投影，取决于距离） |
| 机械臂重复定位精度 | ±0.05 mm（工业级 6/7 DoF） |
| 标定误差 | < 1 mm（手眼标定，眼在手外） |

### 5.4 吞吐量

| 场景 | 节拍 | 说明 |
|------|------|------|
| 简单分拣（缓存命中） | 3–5 s/件 | 模板复用，跳过 VLM 推理 |
| 常规抓放 | 6–10 s/件 | 含 VLM 决策 + 执行 |
| 复杂装配 | 15–30 s/步 | 多步精密操作，每步闭环确认 |

### 5.5 推理性能

平台采用 **云端 API + 边缘推理** 混合架构，灵活适配不同部署条件：

| 环节 | 方案 | 性能 |
|------|------|------|
| LLM / VLM 推理 | 云端大模型 API（如 Qwen-Max、GPT-4o、Claude 等） | 50–200 tok/s，按需调用 |
| VLA 动作推理 | 边缘设备本地推理（Jetson Orin / 工业 PC） | 30–80 ms/step |
| 私有化部署（可选） | 自建 GPU 集群（如 4×H20/A100） | 100+ tok/s，全量本地化 |

> 初期阶段优先调用成熟的云端大模型 API，降低基础设施成本和运维负担；业务规模上量后可按需切换为私有化部署，平台架构无需更改。

## 六、场景自适应

### 6.1 新场景接入流程

```
Step 1 ─ 场景描述（自然语言）
         "从传送带上识别不同颜色的瓶子，分别放入对应的回收箱"

Step 2 ─ LLM 自动拆解
         → 识别颜色 → 确定目标箱 → 抓取 → 搬运 → 释放

Step 3 ─ VLM 构建初始规则
         → 拍照 → 理解场景 → 推理力控策略 → 生成首次执行计划

Step 4 ─ 首次执行 + 闭环观察
         → 逐步执行，每步拍照+力反馈确认 → 记录全流程

Step 5 ─ 模板缓存 + 技能沉淀
         → 成功方案固化为模板 → 力控参数沉淀 → 后续直接复用

Step 6 ─ 持续优化
         → 失败 case 自动收集 → 大模型分析根因 → 微调策略/模型
```

## 七、典型问题：生产中物料形态变化怎么办？

传统方案依赖固定模板匹配，物料外形一变就得停线重新编程调参。本平台基于大模型语义理解，天然具备形态变化适应能力。

### 场景举例

> 生产线上的瓶子从圆瓶换成了方瓶，或者装箱的盒子尺寸变了。

**传统方案：** 停线 → 工程师重新标定模板 → 调整抓取点和力控参数 → 反复测试 → 恢复生产。耗时数小时到数天。

**本平台：** VLM 自动识别新物料形态，推理新的抓取策略，首次执行闭环确认后缓存模板，全程无需人工干预。

### 处理流程

```
瓶子形态变化
    │
    ▼
VLM 自动感知："方形玻璃瓶，比之前的圆瓶更宽，表面平整"
    │
    ▼
推理新策略：平面夹持（非包裹式）、夹爪角度 0°、抓取力 4N
    │
    ▼
首次执行 + 闭环确认（~1-2s 推理 + 拍照验证）
    │
    ├── 成功 → 缓存新模板，后续直接复用
    │
    └── 失败 → 力反馈触发重新观察 → 自动调整策略 → 重试
```

### 不同变化程度的应对

| 变化程度 | 示例 | 处理方式 | 停线时间 |
|----------|------|----------|----------|
| 小变化 | 同类瓶子换尺寸（500ml→330ml） | VLM 自动适应，无需干预 | **0** |
| 中等变化 | 换品类（圆瓶→方瓶） | VLM 首次推理建立新模板 | **0** |
| 大变化 | 全新产品线上线 | 自然语言描述新规则，几次试抓验证 | **< 5 分钟** |

### 为什么能做到？

核心区别在于：传统视觉方案依赖 **模板匹配**（模板变了就废了），本平台依赖 **语义理解**（VLM 理解的是"这是一个瓶子，要放进那个箱子"这个语义，不受具体外形约束）。

同理，容器/盒子变化也一样处理 —— VLM 会重新理解目标区域的位置、尺寸和开口方向，自动调整放置点和释放高度。

## 九、核心优势

| # | 优势 | 说明 |

| 1 | **场景无关** | 不针对特定任务编程，新场景通过自然语言描述即可接入 |
