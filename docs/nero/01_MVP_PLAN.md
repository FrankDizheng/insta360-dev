# NERO MVP 总览

## 1. MVP 的目标

NERO 到货后的第一阶段，不做“全自动复杂具身智能系统”，只做一个能稳定演示的平台最小闭环。

MVP 成功标准：

1. 笔记本可以稳定连上 NERO，完成状态读取、使能、回零、基础运动、夹爪控制。
2. 笔记本可以稳定获取现场相机画面。
3. 一个服务能把“图像 + 任务描述”送给 VLM，拿回结构化动作结果。
4. 一个执行器能把结构化动作翻译成 NERO 的可执行动作。
5. 能完成一个简单 demo，并保留日志和图像，便于复盘。

## 2. 不做什么

MVP 阶段不做以下内容，否则会显著拖慢上线：

- 不做复杂多物体长链规划
- 不做高频闭环视觉伺服
- 不做 MIT 力矩控制
- 不做灵巧手复杂控制
- 不做完整数据飞轮和自动微调
- 不把仿真策略一比一迁移到真机

这些都可以在 MVP 跑通后逐步加。

## 3. 推荐的 MVP Demo

### Demo A: 指定物体抓取并放到指定区域

输入：

- 一张现场画面
- 一句任务描述，例如“把桌上的红色物体抓起来，放到右侧放置区”

输出：

- 识别目标
- 给出抓取目标与放置目标
- 执行固定模板动作序列

动作序列：

1. 移动到预抓取位
2. 下探
3. 夹爪闭合
4. 抬升
5. 移动到放置位
6. 下探
7. 张开夹爪
8. 返回安全位

这个方案最稳，因为：

- VLM 只负责“看懂”和“选目标”
- 真机执行仍然走确定性动作模板
- 风险和调试成本都可控

### Demo B: 人说一句话，系统执行单步搬运

输入：

- 文本指令，例如“把左边那个瓶子拿到框里”

输出：

- LLM / VLM 联合生成一个单次操作意图
- NERO 完成一次 pick-and-place

这个 demo 比 Demo A 更好看，但底层还是同一个执行模板。

## 4. 系统分层

建议把 NERO 集成按五层拆开：

### Layer 0: 设备与通信层

- NERO 上电
- CAN 激活
- Python SDK 或 ROS2 驱动连接成功
- 状态反馈可读

### Layer 1: 执行控制层

- 关节运动
- 末端位姿运动
- 回零、急停、使能、停止
- 夹爪开合

### Layer 2: 视觉采集层

- USB / 网口相机取流
- 单帧抓图
- 标定与工作区裁剪

### Layer 3: 决策层

- 提示词模板
- VLM 调用
- 动作协议解析
- 简单状态机

### Layer 4: Demo 编排层

- 一个任务入口
- 一个执行循环
- 日志、图像、结果保存

### Layer 5: Case 与学习层

- 每一步执行都落成 case
- 支持人工复核与纠偏
- 先由 Copilot / Cursor 等人工智能编程工具辅助完成“人类决策”
- 后续再逐步替换成真正的大模型自动分析与批量学习

这层非常重要，因为它体现的正是你们之前的平台设计思想：

- 先把闭环搭起来
- 再把过程数据沉淀下来
- 再从 batch case 中提炼策略和模板
- 最终走向自适应优化，而不是一次性写死逻辑

## 5. 为什么笔记本方案可行

你刚才给出的判断与实际情况一致，MVP 阶段完全可以用笔记本做现场开发。

原因很直接：

- NERO 控制本身不重，重的是推理
- 官方 `pyAgxArm` 支持 Ubuntu + Python，NERO 可通过 CAN 直接控制
- 官方 `agx_arm_ros` 支持 ROS2 Humble / Jazzy，NERO 也已在驱动中列出
- 本仓库现有 `api_server.py` + `hardware_agent.py` 已经具备“边端采集 / 服务决策 / 执行动作”的骨架

推荐运行形态：

- 笔记本：连 NERO、连相机、跑 agent、跑日志
- 云端或远程 4090：跑 VLM 服务
- 网络：HTTP API 调用推理服务

## 6. MVP 的真实技术路线

### 路线 1: 最稳妥

- VLM 只做识别和语义判断
- 真机动作由固定模板执行

优点：最快落地。

### 路线 2: 再进一步

- VLM 输出结构化动作
- 本地状态机选择动作并调用 NERO

优点：更像“智能决策平台”。

### 路线 2.5: 当前最适合你们的过渡态

- 系统先完整记录 case
- 当前由人工借助 Copilot / Cursor 充当“外部大模型”做复核或纠偏
- 把人工修正过的 case 作为后续自动化策略的训练和验证样本

这条路线很现实，因为它允许你们在没有最终大模型的情况下，先把平台工作流和数据链路建起来。

### 路线 3: 暂不建议作为首周目标

- 图像直接到 VLA 或连续视觉闭环
- 动态修正轨迹

原因：真机安全性、调试难度、现场耗时都明显更高。

## 7. 到货后一周内的目标

### Day 1

- 上电与通信打通
- 状态读取
- 单关节 / MoveJ / MoveP 基础控制
- 夹爪控制

### Day 2

- 相机取流
- 工作区和安全区确认
- 固定点位抓放脚本跑通

### Day 3

- VLM 服务联通
- 图像到动作协议跑通
- 做一次半自动 demo

### Day 4-5

- 串成完整单物体 demo
- 做日志、可视化、异常处理
- 准备对外演示版本
